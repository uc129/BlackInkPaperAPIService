# Database Seed Scripts

Run the scripts in this order:

1. `00_ClearTables.sql`
2. `10_SeedIdentity.sql`
3. `20_SeedCatalog.sql`
4. `30_SeedCommerce.sql`
5. `40_EnsureAutoGeneratedPrimaryKeys.sql`

Or run [`RunAllSeeds.sql`](/Users/utkarshchaudhary/RiderProjects/BlackInkPaperAPIService/Infrastructure/Persistence/Seeds/RunAllSeeds.sql) in SQLCMD mode.

Notes:

- These scripts are written for local/dev seeding and use deterministic IDs where possible.
- [`RunAllSeeds.sql`](/Users/utkarshchaudhary/RiderProjects/BlackInkPaperAPIService/Infrastructure/Persistence/Seeds/RunAllSeeds.sql) uses `:r` includes, so it must be executed with SQLCMD mode enabled.
- The scripts are defensive around a few schema differences that exist between the cached datasource metadata and the repository SQL:
  - `ArtSpecifications`
  - `ProductImages`
  - `Map_ProductTags`
  - `ProductVariantOptions`
- `10_SeedIdentity.sql` seeds usable role/user relationships, but the `PasswordHash` values are placeholders. If you need a real login workflow, keep using the existing C# identity seeder in [`SeedIdentityTables.cs`](/Users/utkarshchaudhary/RiderProjects/BlackInkPaperAPIService/Infrastructure/Persistence/Data Seed/SeedIdentityTables.cs) or replace the hashes with values generated by ASP.NET Identity.
- `40_EnsureAutoGeneratedPrimaryKeys.sql` is intentionally conservative:
  - it verifies the integer `Id` primary keys used by app tables are identity-backed
  - it explains the non-applicable cases (`Users`, `Roles`, `TokenBlacklist`, and composite-key mapping tables)
  - SQL Server can't directly alter an existing column to add `IDENTITY`, so the script prints the migration pattern required for any table that isn't already identity-backed
